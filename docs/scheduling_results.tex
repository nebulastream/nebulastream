\section{Impact of Work Dealing}
In this experiment, we explore the degree of freedom for pushing work to other threads. We want to answer the research question

\begin{tcolorbox}[colback=blue!5, colframe=blue!40!black, title=Research Questions]
    \textbf{RQ:} If a task creates successors, where should this be placed?
\end{tcolorbox}

\subsection{Variants}
In this section, we evaluate four work-dealing scheduling strategies for the query engine's thread pool.
In our SPE, we distinguish between two queues.
The \textit{Admission Queue} is the entry point for external task submission, like data sources. It is bounded and thus implements backpressure by sources. 
The \textit{Internal Queue} is the unbounded function that manages internal tasks, thus work that circulates between the pipeline stages within the engine. 
In particular, if tasks generated other tasks as their successor, for example, the creation of a window result as the first pipeline requires the generation of the window on the window end as a second pipeline.
In general, thread wakeup is implemented via semaphores (see below for more details on the different variants). The main idea is that workers first pick tasks from their internal queue before reaching out to the admission queue. The rationale behind this is to better finish what is currently in-flight before accepting new work.
Please note that the signaling of a sempaphore and the taking of a tasks are not the same. Although a semaphore signal goes to thread X, any worker that happens to check the admission queue can pick up the task. So thread Y might grab a task that thread X was signaled for — that's fine, thread X will just find nothing and block again. As a reesult, the signaled thread isn't guaranteed to be the one that processes the task-


\subsubsection{Single-Queue Strategy}
\textit{GLOBAL\_QUEUE}
All worker threads share a single lock-free MPMC queue (Folly \texttt{MPMCQueue}).
Both admission tasks from sources and successor tasks from pipeline execution are placed into this shared queue.
Workers block on a shared semaphore and dequeue tasks in FIFO order. First it checks the internal queue and if empty the admission queue. 

\textit{Advantages:} Perfect load balancing as no worker can idle while tasks are available.
Simple implementation with no configuration parameters and no priority yet.

\textit{Disadvantages:} Every enqueue and dequeue operation contends on the same atomic state.
At high thread counts, cache line bouncing on the shared queue's head/tail pointers becomes the bottleneck.
No data locality as a successor task is likely processed by a different thread than the one that produced it, requiring the data to be fetched from a remote L2/L3 cache.

\subsubsection{Per-Thread Queue Strategies}
The following strategies assign each worker thread its own unbounded internal queue (Folly \texttt{UMPMCQueue}).
The threads are either woken up by adding a task to the admission queue (thus the current task get selected by round robin) or if a task is added to the internal queue. The sequence is the same, first look into the internal queue and only if this is empty, into the admission queue.

Note that the scheduling strategy's selection policy is \emph{not} applied to admission tasks; it only governs the dispatch of internal (successor) tasks, which are placed directly into a target thread's per-thread queue. We implement the following strategies. They mainly either optimize for maximum locality, thus tasks stay on the same thread vs. optimizing for even load, thus scheduling tasks to other thread to avoid overloading.

\texttt{PER\_THREAD\_ROUND\_ROBIN:}.
Successor tasks are assigned to worker queues in rotating order using an atomic counter:
\[
    \text{target} = \texttt{counter.fetch\_add(1)} \mod N
\]
where $N$ is the number of worker threads.
This provides $O(1)$ dispatch with a single relaxed atomic operation.

\textit{Advantages:} Even distribution of tasks across all threads.
Minimal dispatch overhead (one atomic increment).

\textit{Disadvantages:} No awareness of queue load and thus a slow thread accumulates a backlog while fast threads idle.
In addition, no data locality as successor tasks are distributed away from the producing thread.

\texttt{PER\_THREAD\_SMALLEST\_QUEUE:}
Successor tasks are assigned to the worker queue with the fewest pending tasks.
Queue sizes are tracked via per-queue \texttt{atomic<int64\_t>} counters that are incremented on enqueue and decremented on dequeue, using \texttt{memory\_order\_relaxed}.
The dispatch scans all $N$ counters to find the minimum:
\[
    \text{target} = \arg\min_{i \in [0,N)} \texttt{approxSize}[i]
\]

\textit{Advantages:} Adaptive load balancing and thus threads with shorter queues receive more work.

\textit{Disadvantages:} $O(N)$ dispatch cost (linear scan of all queue sizes).
Approximate sizes may be stale under high concurrency, leading to suboptimal choices.
Multiple threads may simultaneously select the same ``smallest'' queue, leading to a temporary imbalance.
Finally, no data locality is exploited.

\texttt{PER\_THREAD\_CHOOSE\_TWO:}
Successor tasks are assigned to the shorter of two randomly sampled queues (the ``power of two choices'' technique~\cite{mitzenmacher2001power}):
\[
    a, b \sim \text{Uniform}(0, N), \quad \text{target} =
    \begin{cases}
        a & \text{if } \texttt{approxSize}[a] \leq \texttt{approxSize}[b] \\
        b & \text{otherwise}
    \end{cases}
\]
A thread-local \texttt{std::mt19937} random number generator avoids shared state.

\textit{Advantages:} $O(1)$ dispatch with only two atomic reads.
Provably achieves near-optimal load balancing as the maximum queue length is $O(\log \log N)$ compared to $O(\log N)$ for random single-choice and $O(1)$ for full-scan~\cite{mitzenmacher2001power}.
No shared dispatch state (thread-local RNG).

\textit{Disadvantages:} Non-deterministic assignment.
Slightly less balanced than \texttt{SMALLEST\_QUEUE} in low-contention scenarios and no data locality is exploited.

\texttt{PRODUCER\_LOCAL:}
A worker thread that produces a successor task places it on \emph{its own} per-thread queue, bypassing the dispatch strategy entirely.
Tasks from external sources (admission queue) and non-worker contexts (e.g., the delayed task submitter, which re-submits deferred tasks after a timeout) still use the underlying strategy's dispatch policy.

\textit{Advantages:} Optimal data locality, as after processing a tuple buffer through a pipeline stage, the data is hot in the producing thread's L1/L2 cache. However, if the distance between the current task and the new tasks that are enqueued at the end of the queue is too large, for example, the task between both evicts the cache, this is maybe less beneficial. This would happen in highly loaded task queues.
By keeping the successor task (next pipeline stage) on the same thread, we avoid cache misses from cross-thread data migration.
Finally, there is zero dispatch overhead for internal tasks (no atomic operations, no queue selection).

\textit{Disadvantages:} Can cause load imbalance if one thread produces more successors than others, as all successor work accumulates on a single thread's queue.
This can be mitigated by combining it with work stealing.

\texttt{PER\_THREAD\_ADAPTIVE.}
An adaptive strategy that dynamically switches between local keeping and distributed dispatch based on the producer's queue pressure.
When a worker produces a successor task, it checks its own approximate queue size:
\[
    \text{target} =
    \begin{cases}
        \text{producer} & \text{if } \texttt{approxSize}[\text{producer}] \leq 1 \quad \text{(queue shrinking/empty)} \\
        \text{choose-two}(a, b) & \text{if } \texttt{approxSize}[\text{candidate}] < \texttt{approxSize}[\text{producer}] \\
        \text{producer} & \text{otherwise (no less-loaded thread found)}
    \end{cases}
\]
When the producer's queue is empty or has at most one pending task, the successor stays local for cache locality (like \texttt{PRODUCER\_LOCAL}).
When the queue is growing (size $> 1$), the strategy samples two random queues via choose-two and offloads the task to the less loaded one---but only if that candidate is actually less loaded than the producer's queue.
If no better candidate is found, the task stays local.

\textit{Advantages:} Combines the cache locality of \texttt{PRODUCER\_LOCAL} (when the thread is keeping up with its work) with the load balancing of \texttt{CHOOSE\_TWO} (when the thread is falling behind).
The transition is automatic and requires no tuning---the queue size acts as a natural backpressure signal.
Zero overhead in the common case (queue empty, task stays local). In the future, more complex switching algorithms that take more data points, like the load over the last n ms into account, would be possible.

\textit{Disadvantages:} Slightly higher dispatch cost than \texttt{PRODUCER\_LOCAL} due to the queue size check on every dispatch.
The threshold ($\leq 1$) is a heuristic---a thread may keep tasks local even when offloading would improve overall throughput, or offload too eagerly when the queue is temporarily at size~2.
Our experiments show that \texttt{ADAPTIVE} performs comparably to \texttt{PRODUCER\_LOCAL} at low thread counts and slightly better under high load, but does not outperform \texttt{GLOBAL\_QUEUE} on workloads with few successor tasks.

\subsection{Experimental Results: Per-Thread Scheduling Strategies}
\label{sec:scheduling-results}

We evaluate the six work-dealing strategies on five workloads using the Yahoo Streaming Benchmark (YSB) with 50 million tuples (4.8\,GB) and 10,000 distinct campaign IDs.
All experiments run on a dual-socket server with 2 NUMA nodes (32 logical CPUs each, 16 physical cores with hyperthreading) and 512\,GB RAM.
For thread counts $\leq 32$, processes are pinned to NUMA node~0 via \texttt{numactl --cpunodebind=0 --membind=0}.
At 64 threads, both NUMA nodes are used without pinning.

\subsubsection{Stateless Workload (Filter Only)}

The stateless workload applies a predicate (\texttt{WHERE event\_type < 1}) with approximately 80\% selectivity.
This is a single-pipeline query that produces \emph{no successor tasks}, thus each source buffer is processed in a single pipeline execution through filter and checksum sink.
The only scheduling-relevant contention is on the admission queue (source $\rightarrow$ workers).

The key observations are as follows.
% 
\begin{itemize}
    \item \textbf{At 1--8 threads, all strategies perform identically} (within 3\%), with \texttt{PRODUCER\_LOCAL} and \texttt{GLOBAL\_QUEUE} slightly ahead.
    At low thread counts, queue contention is negligible, so the dispatch policy has minimal impact.

    \item \textbf{\texttt{GLOBAL\_QUEUE} dominates at 16--64 threads}, achieving 564\,MB/s at 16T (7.8$\times$) and peaking at 593\,MB/s at 32T (8.2$\times$).
    Since this workload produces no successor tasks, the per-thread queue strategies add dispatch overhead without any locality benefit.
    The single shared queue provides perfect load balancing with no dispatch cost for internal tasks (there are none).

    \item \textbf{Per-thread strategies diverge at 16T and above.}
    \texttt{ROUND\_ROBIN} drops to 395\,MB/s at 16T and collapses to 191\,MB/s at 64T.
    The round-robin counter becomes a contention point when many threads increment it simultaneously, and the even distribution can leave some queues overfull while others are empty if thread execution speeds vary.
    \texttt{SMALLEST\_QUEUE} and \texttt{CHOOSE\_TWO} fare better (400--412\,MB/s at 16T) because they adapt to load imbalance, but still trail \texttt{GLOBAL\_QUEUE} due to the overhead of maintaining per-thread queues for a workload that does not benefit from them.

    \item \textbf{\texttt{PRODUCER\_LOCAL} offers no advantage over \texttt{GLOBAL\_QUEUE}} on this workload.
    Since there are no successor tasks, the producer-local policy is never triggered — every task comes from the admission queue.
    The 477\,MB/s at 32T (vs.\ 593 for \texttt{GLOBAL\_QUEUE}) reflects the overhead of the per-thread queue infrastructure without any benefit.

    \item \textbf{The 32T $\rightarrow$ 64T throughput drop} is caused by two factors:
    (1)~at 32T, all threads share 16 physical cores via hyperthreading, reducing per-thread throughput;
    (2)~at 64T, NUMA pinning is disabled, introducing cross-node memory access latency.
    \texttt{GLOBAL\_QUEUE} handles this best (388\,MB/s) because its single shared queue naturally load-balances across both NUMA nodes.
\end{itemize}

\begin{itemize}
    \item \textbf{Scaling is limited to 2.3$\times$}, with \texttt{GLOBAL\_QUEUE} achieving the peak of 179\,MB/s at 32T.
    The aggregation operator's hash table and window state management are the bottleneck, not the scheduling strategy.
    Adding more threads provides diminishing returns because the per-tuple cost of hash table lookups and state updates dominates.

    \item \textbf{All strategies perform within 10--15\% of each other} at every thread count.
    This confirms that for compute-bound stateful queries, the choice of work-dealing scheduling strategy is secondary to operator efficiency.

    \item \textbf{\texttt{GLOBAL\_QUEUE} wins at most thread counts} (2T, 4T, 8T, 32T, 64T), consistent with the stateless results.
    Its advantage is most pronounced at 32T (179 vs.\ 152--171 for per-thread strategies), where the single shared queue avoids the per-thread overhead that provides no benefit for this workload.

    \item \textbf{The throughput dip at 16T} (133--139\,MB/s, below 8T at 141--147\,MB/s) is caused by increased contention on the shared aggregation hash table.
    With 16 threads all performing concurrent hash lookups and window state updates, synchronization overhead temporarily exceeds the parallelism gain.
    At 32T, throughput recovers as hyperthreading allows more threads to overlap memory stalls with computation.

    \item \textbf{\texttt{SMALLEST\_QUEUE} performs worst} at low thread counts (127\,MB/s at 4T vs.\ 142 for \texttt{GLOBAL\_QUEUE}).
    The $O(N)$ linear scan of all queue sizes adds measurable overhead when tasks are short (stateful tasks spend most time in the aggregation operator, leaving little room for scheduling overhead to hide).

    \item \textbf{\texttt{PRODUCER\_LOCAL} provides no measurable benefit} despite the stateful query producing window emission successor tasks.
    The reason is that window emissions are too rare (1,667 across 50M tuples) to create meaningful cache locality advantages.
\end{itemize}

    Cross-NUMA memory access dominates; only strategies with strong locality (\texttt{PRODUCER\_LOCAL}) or fair distribution (\texttt{GLOBAL\_QUEUE}) survive.
\end{itemize}

\subsubsection{Mixed Concurrent Workload}

Both queries share the same worker thread pool and compete for scheduling resources.
The filter query processes buffers quickly and produces no successor tasks, while the aggregation query is compute-heavy and produces $\sim$5 million window emission successor tasks.
This creates asymmetric load: workers processing filter buffers finish quickly and idle, while workers processing aggregation buffers are busy and produce successor tasks that must be dispatched.

    \item \textbf{Scaling reaches 4.8$\times$ at 32T} (\texttt{PRODUCER\_LOCAL} and \texttt{GLOBAL\_QUEUE} both at 41\,s), the highest speedup across all tested workloads.

    \item \textbf{\texttt{ADAPTIVE} fails to match \texttt{PRODUCER\_LOCAL}} (49\,s vs.\ 45\,s at 8T).
    The mixed workload creates bursty queue behavior: aggregation successor tasks arrive in clusters (window emissions), temporarily inflating the queue size above the threshold and triggering offloading.
    A purely local strategy (\texttt{PRODUCER\_LOCAL}) handles this better by never offloading, letting the worker drain its burst naturally.

    \item \textbf{The mixed workload is the strongest differentiator} among all tested workloads.
    The 10\% gap between \texttt{PRODUCER\_LOCAL} and \texttt{GLOBAL\_QUEUE} at 8T is larger than the gaps observed in the high-frequency window (18\% in throughput, but on a different metric) and join (7\%) workloads, confirming that concurrent query execution with asymmetric workloads is the scenario where work-dealing strategy choice matters most.
\end{itemize}

\subsubsection{Summary}
We analyze the impact of work-dealing on throughput across five workloads with increasing successor task frequency and concurrency:


\begin{itemize}
    \item \textbf{The optimal strategy depends on successor task frequency and query concurrency.}
    When successor tasks are rare or absent (stateless filter, 30s window), \texttt{GLOBAL\_QUEUE} dominates due to its perfect load balancing and zero dispatch overhead.
    When successor tasks are frequent (10\,ms window, joins, or concurrent mixed workloads), \texttt{PRODUCER\_LOCAL} wins by 10--18\% through cache locality.

    \item \textbf{\texttt{PRODUCER\_LOCAL} is the best overall strategy for multi-pipeline and concurrent workloads.}
    It leads on the high-frequency window workload (+18\% at 8T), the join workload (+7\% at 32T), and the mixed concurrent workload ($-$10\% wall time at 8T).
    The benefit is consistent across three fundamentally different scenarios (single-source aggregation, multi-source join, and concurrent heterogeneous queries), confirming that cache locality for successor tasks is the key advantage.

    \item \textbf{\texttt{GLOBAL\_QUEUE} is the safest default when workload characteristics are unknown.}
    It wins or ties on the stateless and stateful workloads at most thread counts, and remains competitive on successor-heavy workloads.
    Its simplicity (no configuration parameters) and natural load balancing make it robust.
    It is also the only strategy that degrades gracefully at 64T (cross-NUMA).

    \item \textbf{Per-thread dispatch strategies (\texttt{ROUND\_ROBIN}, \texttt{CHOOSE\_TWO}, \texttt{SMALLEST\_QUEUE}) add overhead without sufficient benefit.}
    They trail both \texttt{GLOBAL\_QUEUE} and \texttt{PRODUCER\_LOCAL} on all five workloads.
    The dispatch cost and loss of cache locality for successor tasks outweigh any load balancing advantage.
    On the mixed workload at 64T, they collapse to 170--175\,s vs.\ 98\,s for \texttt{GLOBAL\_QUEUE}.

    \item \textbf{\texttt{ADAPTIVE} does not outperform \texttt{PRODUCER\_LOCAL}} on any tested workload.
    Bursty successor task arrivals (window emissions, join output) temporarily inflate queue sizes above the threshold, triggering offloading that negates the locality benefit.
    A purely local strategy handles bursts better by letting the worker drain them naturally.

    \item \textbf{The work-dealing strategy matters most under two conditions:}
    (1)~high successor task frequency (10\,ms windows, joins), where the gap between best and worst strategy reaches 18\%;
    (2)~concurrent query execution with asymmetric workloads, where the gap reaches 10\% even at moderate thread counts.
    At 1--2 threads, all strategies perform within 3\%.

    \item \textbf{Cross-NUMA execution (64T) severely degrades per-thread strategies.}
    On the mixed workload, per-thread dispatch strategies are 75\% slower than \texttt{GLOBAL\_QUEUE} at 64T.
    Only \texttt{PRODUCER\_LOCAL} remains competitive cross-NUMA (115\,s vs.\ 98\,s), because keeping tasks local avoids cross-node data migration.
    NUMA-aware scheduling remains the most impactful optimization opportunity beyond work-dealing strategy choice.
\end{itemize}
