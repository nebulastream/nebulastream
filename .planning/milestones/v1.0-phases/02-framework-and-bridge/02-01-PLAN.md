  <action>
Create `bridge.rs` with:

1. `BridgeMessage` struct: `{ origin_id: u64, buffer: TupleBufferHandle, permit: tokio::sync::OwnedSemaphorePermit }`.

2. `BridgeHandle` struct: `{ sender: async_channel::Sender<BridgeMessage>, thread: Option<std::thread::JoinHandle<()>> }`.

3. Type aliases for the emit callback: `type EmitFnPtr = unsafe extern "C" fn(*mut std::ffi::c_void, u64, *mut ffi::TupleBuffer) -> u8;` -- this matches the function pointer + void* context pattern from the user decision.

4. **Per-source emit registry (CRITICAL for multi-source correctness):**

   `EmitEntry` struct: `{ emit_fn: EmitFnPtr, emit_ctx: *mut std::ffi::c_void }`.
   `unsafe impl Send for EmitEntry {}` and `unsafe impl Sync for EmitEntry {}` -- the void* context pointer is guaranteed valid for the source's lifetime by C++ (EmitContext is owned by TokioSource which outlives the Rust source task).

   `static EMIT_REGISTRY: LazyLock<DashMap<u64, EmitEntry>>` -- per-source emit callback registry. Each source registers its own EmitFunction + context when spawned.

   Free functions:
   - `register_emit(source_id: u64, emit_fn: EmitFnPtr, emit_ctx: *mut std::ffi::c_void)` -- inserts into EMIT_REGISTRY
   - `unregister_emit(source_id: u64)` -- removes from EMIT_REGISTRY

   **Why registry, not per-message:** The bridge is shared across all sources (user decision). Each source has a distinct EmitFunction (per-query state). The bridge must dispatch each message to the correct source's emit callback. A registry keyed by source_id is the simplest correct approach. Per-message function pointers would bloat BridgeMessage and complicate the channel.

5. `static BRIDGE: OnceLock<BridgeHandle>` -- process-global, lazily initialized per user decision.

6. `ensure_bridge() -> &'static async_channel::Sender<BridgeMessage>` -- takes NO emit callback parameters (emit dispatch is handled by EMIT_REGISTRY). Initializes bridge on first call via OnceLock. Creates `async_channel::bounded(64)` (small fixed capacity per user decision). Spawns a named `std::thread` ("nes-bridge") running `bridge_loop`.

7. `bridge_loop(receiver)` -- calls `receiver.recv_blocking()` in a loop. For each message:
   - Looks up `EMIT_REGISTRY.get(&msg.origin_id)` to find the source's emit callback
   - If found: calls `(entry.emit_fn)(entry.emit_ctx, msg.origin_id, buffer raw ptr)`
   - If NOT found (source already unregistered -- race during shutdown): logs a warning via tracing, drops the message (permit and buffer are dropped, releasing resources)
   - After the call returns, permit and buffer are dropped (permit releases semaphore, buffer releases C++ refcount)
   - On channel close (Err from recv_blocking), thread exits cleanly

8. For testing: create a `#[cfg(test)]` version using mock emit entries that record calls into `Arc<Mutex<Vec<(u64, u64)>>>` (source_id, origin_id pairs). Test multi-source dispatch by registering two different mock callbacks and verifying messages route correctly.

Add `pub mod bridge;` to lib.rs and re-export `BridgeMessage`, `BridgeHandle`, `EmitFnPtr`, `register_emit`, `unregister_emit`.

Note: The actual `bridge_loop` with the C-style emit function pointer will require `unsafe`. Use `// SAFETY:` comments per project convention. For tests, use a test-only factory function that takes a Rust closure instead of function pointer.
