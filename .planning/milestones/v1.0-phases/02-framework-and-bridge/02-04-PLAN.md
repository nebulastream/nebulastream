Fix the BLOCKER gap where bridge_loop passes a null buffer pointer to dispatch_message instead of the actual TupleBuffer raw pointer from the BridgeMessage.

Purpose: Without this fix, C++ bridge_emit receives nullptr as buffer_ptr and crashes on rawBuffer->retain() (null pointer dereference). This is the single defect preventing the async-to-C++ buffer pipeline from functioning.

Output: A working bridge_loop that extracts the raw TupleBuffer pointer from BridgeMessage.buffer and passes it to the emit callback. After this fix, buffers flow from Rust source tasks through the bridge thread into the C++ Folly MPMC task queue.

uint8_t bridge_emit(void* context, uint64_t origin_id, void* buffer_ptr) {
    auto* rawBuffer = static_cast<NES::TupleBuffer*>(buffer_ptr);
    rawBuffer->retain();  // <-- crashes if buffer_ptr is null
    NES::TupleBuffer buffer(*rawBuffer);
    auto result = ctx->emitFunction(OriginId(origin_id), Data{std::move(buffer)}, ...);
