<design_rationale>
## Why move runtime init INTO the bindings crate (not the other way)

The dependency graph is: runtime -> bindings (one-way). Adding bindings -> runtime
would create a cycle. cxx requires extern "Rust" functions to be defined in the
same crate as the #[cxx::bridge] module -- this is a compile-time constraint, not
a link-time one.

Therefore the OnceLock<Runtime> and init_source_runtime() must live in the bindings
crate. The runtime crate's source_runtime() accessor must delegate to the bindings
crate's public function. This preserves the one-way dependency edge.

The runtime crate's existing tests for init/double-init should be updated to call
the bindings crate's function (or remain as integration-style tests that test the
public API via the re-export path).
</design_rationale>
