# Roadmap: Tokio-Based Rust Sources for NebulaStream

## Overview

Three phases build the async Rust source framework from the ground up. Phase 1 establishes the safe FFI type system and the shared Tokio runtime — everything else depends on getting this right. Phase 2 wires the full pipeline: bridge thread, backpressure in both directions, per-source semaphore, and the `AsyncSourceHandle` that plugs into the existing C++ engine without modifying it. Phase 3 validates the complete framework via a generator source and end-to-end integration tests that exercise the critical invariants (backpressure stability, graceful shutdown, zero permit leaks).

## Phase Details

### Phase 1: FFI Foundation
**Goal**: Safe Rust ↔ C++ interop exists and the shared Tokio runtime is initialized — all higher-level code can be built on this without revisiting FFI safety
**Depends on**: Nothing (first phase)
**Requirements**: FFI-01, FFI-02, FFI-03, FFI-04, FWK-01, FWK-03
**Success Criteria** (what must be TRUE):
  1. `TupleBufferHandle` and `BufferProviderHandle` compile with correct `Clone` (calls `retain()`) and `Drop` (calls `release()`) and pass a refcount assertion test that verifies no premature release across a clone-drop cycle
  2. A unit test confirms that any Rust panic inside an FFI-exposed function is caught by `catch_unwind` and does not abort the process
  3. The shared Tokio runtime initializes exactly once (via `OnceLock`) when given a `WorkerConfiguration` thread count, and subsequent init calls are no-ops
  4. `SourceContext` compiles with `allocate_buffer()` and `emit(buffer).await` signatures and exposes no raw pointers or C++ types to callers
**Plans:** 3 plans (2 complete, 1 gap closure)

### Phase 2: Framework and Bridge
**Goal**: The full async-to-C++ pipeline is wired — a Rust source task can emit a buffer through the bridge thread into the Folly MPMC task queue, with backpressure in both directions and graceful shutdown
**Depends on**: Phase 1
**Requirements**: FWK-02, FWK-04, FWK-05, FWK-06, FWK-07, BRG-01, BRG-02, BRG-03, BRG-04
**Success Criteria** (what must be TRUE):
  1. A Rust source task can call `ctx.emit(buf).await` and the buffer arrives in the Folly MPMC task queue via the bridge thread, with the semaphore permit released via the `OnComplete` callback
  2. When the inflight buffer count reaches `defaultMaxInflightBuffers`, additional `emit()` calls yield (not block a Tokio worker thread) until a permit is released
  3. A C++ `BackpressureListener` signal causes an awaiting Rust source task to resume without blocking any Tokio worker thread
  4. Calling `stop()` on `AsyncSourceHandle` signals the `CancellationToken`, waits for the source task to exit, drains the bridge channel, and leaves zero leaked semaphore permits
  5. `RunningSource` can start and stop an async Rust source via the existing `SourceHandle` interface with no changes to `RunningSource` or `QueryEngine`
**Plans:** 4/4 plans complete

### Phase 3: Validation
**Goal**: The framework is proven correct end-to-end via a concrete source implementation and tests that exercise every critical invariant
**Depends on**: Phase 2
**Requirements**: VAL-01, VAL-02
**Success Criteria** (what must be TRUE):
  1. `GeneratorSource` (async sleep loop emitting incrementing `u64` values) compiles as a complete `AsyncSource` implementation using only `ctx.allocate_buffer()` and `ctx.emit(buf).await`
  2. An end-to-end integration test registers `GeneratorSource`, runs a pipeline, and verifies that emitted buffers arrive at the sink in the correct order with the correct values
  3. A backpressure regression test artificially slows the bridge and confirms memory usage stays stable (no unbounded buffer accumulation) under sustained emission
  4. A shutdown race test issues `stop()` while the source is mid-emit and verifies the test completes with zero leaked semaphore permits
**Plans:** 2 plans
