  </files>
  <action>
    This task has two major parts: getting ANTLR4 parser sources and building them to WASM.

    **Part A: Get ANTLR4-generated C++ sources from AntlrSQL.g4**

    The ANTLR4 code generator is a Java JAR. Rather than requiring Java in the WASM build, pre-generate the C++ parser sources.

    Option 1 (preferred): Check if NES build output already has generated files:
    ```bash
    find /IdeaProjects/nebulastream2 -name "AntlrSQLParser.cpp" -o -name "AntlrSQLParser.h" 2>/dev/null | head -5
    ```
    If found, copy them to `spike-antlr4/generated/`.

    Option 2: Generate from grammar using the ANTLR4 tool:
    ```bash
    # Download ANTLR4 tool JAR (check which version NES uses in vcpkg.json)
    curl -O https://www.antlr.org/download/antlr-4.13.2-complete.jar
    java -jar antlr-4.13.2-complete.jar -Dlanguage=Cpp -visitor -o spike-antlr4/generated nes-sql-parser/AntlrSQL.g4
    ```
    Match the ANTLR4 version to what vcpkg provides (check `vcpkg search antlr4`).

    Option 3: If Java is unavailable, use pip install antlr4-tools or find the generated sources in the NES build tree.

    **Part B: Create stub header for NES-specific grammar includes**

    Create `spike-antlr4/include/Util/DisableWarningsPragma.hpp`:
    ```cpp
    #pragma once
    // Stub for NES-specific warning pragmas -- not needed in WASM build
    #define DISABLE_WARNING_PUSH
    #define DISABLE_WARNING_POP
    #define DISABLE_WARNING(x)
    ```
    This resolves Pitfall 3 from research (grammar @postinclude blocks reference this header).

    **Part C: Create ANTLR4 spike main.cpp**

    Create `spike-antlr4/main.cpp` following the RESEARCH.md Pattern 3 example:
    - Include `<emscripten/bind.h>`, `<antlr4-runtime.h>`, generated parser/lexer headers
    - Implement `parseSql(const std::string& sql)` that:
      1. Creates ANTLRInputStream from sql string
      2. Creates AntlrSQLLexer and CommonTokenStream
      3. Creates AntlrSQLParser and calls parser.statement()
      4. Returns "PARSE_ERROR" if syntax errors > 0, else the parse tree string
    - Expose via EMSCRIPTEN_BINDINGS with `emscripten::function("parseSql", &parseSql)`

    **Part D: Create CMakeLists.txt for spike-antlr4**

    ```cmake
    # Find the ANTLR4 runtime (installed via vcpkg)
    find_package(antlr4-runtime CONFIG REQUIRED)

    # Collect generated parser sources
    file(GLOB ANTLR4_GENERATED_SRCS "generated/*.cpp")

    add_executable(spike-antlr4
        main.cpp
        ${ANTLR4_GENERATED_SRCS}
    )

    target_include_directories(spike-antlr4 PRIVATE
        generated/       # Generated parser/lexer headers
        include/         # Stub headers (DisableWarningsPragma.hpp)
    )

    target_link_libraries(spike-antlr4 PRIVATE antlr4_shared)  # or antlr4_static
    target_link_options(spike-antlr4 PRIVATE -lembind)
    set_target_properties(spike-antlr4 PROPERTIES
        SUFFIX ".js"
        LINK_FLAGS "-sMODULARIZE=1 -sEXPORT_ES6=0 -sENVIRONMENT=node -fwasm-exceptions -sALLOW_MEMORY_GROWTH=1"
    )
    ```
    Use `-fwasm-exceptions` for native WASM exception handling (ANTLR4 uses exceptions -- see Pitfall 4). Use `-sALLOW_MEMORY_GROWTH=1` since ANTLR4 allocates dynamically. If `antlr4_shared` doesn't work (WASM needs static), switch to `antlr4_static` or the raw target name from vcpkg.

    **Part E: Update top-level CMakeLists.txt** to add `add_subdirectory(spike-antlr4)`.

    **Part F: Update vcpkg.json** to add antlr4 dependency:
    ```json
    {
      "name": "nes-wasm-spikes",
      "version-string": "0.1.0",
      "dependencies": ["antlr4"]
    }
    ```

    **Part G: Build**

    ```bash
    cd nes-topology-editor/wasm
    source emsdk/emsdk_env.sh
    emcmake cmake -B build \
      -DCMAKE_TOOLCHAIN_FILE=$VCPKG_ROOT/scripts/buildsystems/vcpkg.cmake \
      -DVCPKG_TARGET_TRIPLET=wasm32-emscripten \
      -DVCPKG_OVERLAY_TRIPLETS=triplets \
      -DVCPKG_CHAINLOAD_TOOLCHAIN_FILE=$EMSDK/upstream/emscripten/cmake/Modules/Platform/Emscripten.cmake
    cmake --build build --target spike-antlr4
    ```

    If vcpkg `antlr4:wasm32-emscripten` fails (Pitfall 6 from research), fall back to building ANTLR4 C++ runtime from source:
    1. Clone antlr4 repo, cd to `runtime/Cpp`
    2. Build with Emscripten: `emcmake cmake -B build && cmake --build build`
    3. Point spike CMakeLists.txt to the built library

    **Compile flags for ANTLR4 warnings**: The ANTLR4 runtime may produce warnings with C++23. Add `-Wno-deprecated-declarations` and `-Wno-deprecated-copy` to the spike target if needed.

  <action>
    **Step 1: Create test file** at `test/test-antlr4.mjs`:
    ```javascript
    import { createRequire } from 'module';
    const require = createRequire(import.meta.url);
    const createModule = require('../build/spike-antlr4/spike-antlr4.js');

    const Module = await createModule();

    // Test 1: Valid NES SQL parses without error
    const validSql = "SELECT * FROM default_logical";
    const validResult = Module.parseSql(validSql);
    console.assert(!validResult.includes("PARSE_ERROR"),
        `Valid SQL should parse: ${validResult}`);
    console.log(`Valid SQL result: ${validResult}`);

    // Test 2: Invalid SQL returns PARSE_ERROR
    const invalidSql = "SELECTFROM broken!!!";
    const invalidResult = Module.parseSql(invalidSql);
    console.assert(invalidResult === "PARSE_ERROR",
        `Invalid SQL should fail: ${invalidResult}`);

    // Test 3: NES-specific SQL with WINDOW clause
    const nesSql = "SELECT key, value FROM default_logical WINDOW TumblingWindow(size = 10)";
    const nesResult = Module.parseSql(nesSql);
    console.assert(!nesResult.includes("PARSE_ERROR"),
        `NES SQL should parse: ${nesResult}`);

    console.log("PASS: ANTLR4 WASM spike");
    ```

    Adjust the SQL examples based on what the AntlrSQL.g4 grammar actually accepts. Check the grammar's `statement` rule entry point. If the grammar uses `query` instead of `statement`, update `main.cpp` accordingly.

    **Step 2: Run the test**:
    ```bash
    cd nes-topology-editor/wasm
    node test/test-antlr4.mjs
    ```

    **Step 3: If tests fail**, debug:
    - If Module.parseSql is undefined: check Embind bindings in main.cpp
    - If valid SQL returns PARSE_ERROR: check which grammar rule is the entry point (might be `query()` not `statement()`)
    - If it crashes: check exception handling flags (`-fwasm-exceptions` vs `-fexceptions`)
    - If memory issues: ensure `-sALLOW_MEMORY_GROWTH=1` is set
